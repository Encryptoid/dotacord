# Heroes Database & `/heroes` Command

## Context

Heroes are currently stored in `data/heroes.json` (a static file) and cached in
`hero_cache.rs` as a `HashMap<i32, String>` (ID → localized name). This approach has
two problems: the file must ship at runtime, and there's no way to store user-maintained
position data.

**This spec replaces `heroes.json` with a `heroes` database table** as the single source
of truth. The JSON file is used once for initial hydration, then removed from the project.
`hero_cache.rs` is deleted and replaced with a per-request `HeroLookup` struct that loads
all heroes from DB once per request (leaderboard, tool call, etc.) and is passed around
as a reference. Heroes get user-maintained position flags toggled via a `/heroes` Discord
command.

**Position flags**: `is_carry`, `is_mid`, `is_offlane`, `is_support` (a hero can have multiple).
Initial hydration seeds `is_carry` and `is_support` from heroes.json roles. `is_mid` and
`is_offlane` default to 0 — users set these via the `/heroes` command.

This is a prerequisite for the `top_winrate_heroes` AI tool.

## Example Usage

```
/heroes Storm Spirit
/heroes stormspirit
```

Opens an interactive panel showing the hero name and four toggle buttons:
- **Carry** (on/off)
- **Mid** (on/off)
- **Offlane** (on/off)
- **Support** (on/off)

A hero must have **at least one** position flag set. Users click to toggle and the DB
updates immediately, but toggling off is blocked if it would leave the hero with no
position flags.

## Flow

Entry: `/heroes <hero_name>` slash command (no admin restriction)
  -> Parse hero_name (case-insensitive, spaces optional: "Storm Spirit" or "StormSpirit")
  -> Look up hero in DB by name match
  -> Show ephemeral panel with hero name + 4 position toggle buttons
  -> ComponentInteractionCollector loop (60s timeout)
  -> On button click: toggle position flag in DB, refresh panel
  -> On timeout: close panel

### 5. Delete hero_cache.rs, add HeroLookup

**Delete**: `src/database/hero_cache.rs`

**Add to**: `src/database/heroes_db.rs`

Per-request lookup struct — loaded once per request from a single DB query, passed as `&HeroLookup`:

```rust
pub struct HeroLookup {
    heroes: HashMap<i32, HeroModel>,
}

impl HeroLookup {
    pub async fn load() -> Result<Self, Error> {
        let all = query_all_heroes().await?;
        let heroes = all.into_iter().map(|h| (h.hero_id, h)).collect();
        Ok(Self { heroes })
    }

    pub fn get_name(&self, hero_id: i32) -> Option<&str> {
        self.heroes.get(&hero_id).map(|h| h.name.as_str())
    }
}
```

### 7. Slash command

**File**: `src/discord/heroes_command.rs`

Follow `server_settings_command.rs` for interactive panel pattern.

- Arg: `hero_name: String` — required
- Hero name matching: case-insensitive, also match with spaces removed
  (e.g., "stormspirit" matches "Storm Spirit")
- Panel shows: hero localized name + 4 position toggle buttons (Carry, Mid, Offlane, Support)
- Buttons use `build_toggle_button` pattern from server_settings_command
- On toggle: update DB, refresh panel
- **Enforce constraint**: if toggling off would leave hero with no position flags set,
  block the toggle (don't update DB, keep button in current state)
- Ephemeral message, 60s timeout

## Notes

- `hero_cache.rs` is deleted — replaced by per-request `HeroLookup` in `heroes_db.rs`
- No global cache — each request loads fresh from DB (single query), never stale
- The `heroes` DB table is the single source of truth for hero names and position classification
- `data/heroes.json` is consumed once by `scripts/gen_heroes_sql.py` to produce
  `sql/hydrate_heroes.sql`, then deleted
- Discord slash commands support max 25 options in autocomplete, so hero_name is a free-text
  string arg, not a select menu (100+ heroes)
- **Invariant**: every hero must have at least one position flag set — never all 0.
  Enforced at hydration (heroes.json prerequisite) and at toggle time (UI constraint)
